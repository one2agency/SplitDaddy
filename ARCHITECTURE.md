# Як працює SplitDaddy: архітектура та логіка

Документ пояснює, **що** робить бот і **яку логіку** в нього закладено — щоб у код
можна було повернутись через місяці й швидко все згадати.

---

## 1. Загальна ідея

Бот живе у **груповому чаті**. Компанія створює «вечірку» (сесію), протягом
вечора додає витрати (хто заплатив і за кого), а в кінці бот рахує **хто кому
винен** мінімальною кількістю переказів.

Ключові рішення:
- **Без слеш-команд** — усі дії через кнопки з українськими підписами + текстовий
  ввід (сума на початку повідомлення). Так не треба перемикати розкладку.
- **Long polling** — бот сам ходить до Telegram; не потрібні порти/домен.
- **Гроші в копійках** (`int`) — жодних похибок плаваючої коми.
- **Privacy Mode = OFF** — інакше бот не бачить звичайних повідомлень (кнопок і
  витрат), а лише слеш-команди.

---

## 2. Потік взаємодії

```
Бота додають у групу ──► показує клавіатуру (welcome)
        │
        ▼
🎉 Нова вечірка ──► створюється сесія з кодом (#BX), натиснувший = організатор
        │
        ▼
Учасники додаються: 🙋 Я в ділі / + @user / автор витрати / тег у витраті
        │
        ▼
Текст «350 коктейлі @ira -я» ──► витрата #BX-n, бот підтверджує (✏️/🗑 кнопки)
        │
        ▼
📊 Підсумок (будь-коли) / ✅ Завершити (лише організатор) ──► хто кому винен
```

### Маршрутизація повідомлень (`handlers.on_text`)
Звичайний текст розбирається за пріоритетом:
1. `видалити BX-3` → видалення витрати;
2. `+ @user` / `- @user` → керування учасниками;
3. повідомлення, що **починається з числа** → витрата (або **редагування**, якщо
   це reply на повідомлення витрати);
4. усе інше (балачки, фото, меми) — **ігнорується**.

Якщо у чаті немає відкритої вечірки, текстові патерни ігноруються мовчки — щоб
бот не шумів у звичайній переписці.

---

## 3. Правила поділу витрати

Платник (payer) — **завжди автор повідомлення**. Бенефіціари визначаються так
(`handlers._resolve_beneficiaries`):

| Ввід | На кого ділиться |
|---|---|
| `1200 піца` (без тегів) | на **всіх** учасників вечірки |
| `350 коктейлі @ira @oleh` | на **відмічених + автора** |
| `800 таксі @ira -я` | лише на **відмічених** (автор виключений через `-я`) |
| `1200 -я` (без тегів) | на всіх, **крім автора** |

- Сума ділиться **порівну**. Залишок копійок від ділення розподіляється по 1 коп.
  **першим** бенефіціарам, тож сума часток точно дорівнює витраті
  (`splitter.split_amount`).
- Тегнуті користувачі **авто-додаються** в учасники.
- Тег `@username` людини, яку бот ще «не знає», не резолвиться (див. §6) — витрата
  не додається, доки та людина не проявиться в чаті.

## 4. Підрахунок балансу й мінімізація переказів

Уся математика — у `bot/splitter.py` (чисті функції, **без** залежностей від
Telegram чи БД; саме тому її легко тестувати).

1. **Баланс** кожного учасника: `net = paid − owed`
   (скільки заплатив мінус сума його часток у всіх витратах).
   Сума всіх `net` завжди = 0.
2. `net > 0` — **кредитор** (йому винні), `net < 0` — **боржник**.
3. **Жадібна мінімізація** (`minimize_transfers`): на кожному кроці беремо
   найбільшого боржника й найбільшого кредитора та гасимо максимально можливу
   суму. Для `k` ненульових балансів виходить щонайбільше `k−1` переказів.

### Приклад (вечірка #BX: Я, Олег, Іра, Макс)
| Витрата | Платник | Текст | Бенефіціари |
|---|---|---|---|
| #BX-1 | Я | `1200 коктейлі` | усі (по 300) |
| #BX-2 | Олег | `600 таксі @ira -я` | лише Іра (600) |
| #BX-3 | Іра | `400 вхід` | усі (по 100) |

Баланси: Я +800, Олег +200, Іра −600, Макс −400 →
**Іра → Я 600, Макс → Я 200, Макс → Олег 200** (3 перекази).

## 5. Модель даних (SQLite)

| Таблиця | Призначення |
|---|---|
| `parties` | вечірки: `code` (унікальний 2–4 символи), `organizer_user_id`, `status` |
| `participants` | учасники вечірки (`UNIQUE(party_id, tg_user_id)`) |
| `expenses` | витрати: `seq` (номер у межах вечірки) + `UNIQUE(party_id, seq)` |
| `expense_shares` | бенефіціари кожної витрати (порядок = розподіл копійок) |
| `known_users` | глобальний довідник `tg_user_id ↔ username/name` для резолву тегів |

### Людські ID витрат
Користувач бачить `#<code>-<seq>` (напр. `BX-3`), а не глобальний `expenses.id`.
`code` — Base32 без схожих символів (немає `0/O`, `1/I`), глобально унікальний.
Завдяки цьому **паралельні вечірки в різних чатах не плутаються**.

### Конкурентність
Записи серіалізуються через `asyncio.Lock`, а складені операції (витрата +
частки, оновлення, видалення) — в одній транзакції (`BEGIN`/`COMMIT`/`ROLLBACK`).

## 6. Обмеження Telegram, які формують дизайн

- **Список учасників групи недоступний** ботам → учасників формуємо явно
  (теги / «Я в ділі» / авто-додавання автора витрати).
- **`@username` → `user_id` бот резолвити не вміє**, доки людина сама щось не
  напише в чат (або не натисне «Я в ділі»). Тому є глобальний `known_users`, який
  наповнюється через middleware на кожне повідомлення.
- **`text_mention`** (тег людини без username) несе `user.id` всередині entity —
  саме тому парсер читає **entities**, а не текст.
- **`request_users`** (нативний пікер) працює **лише в приватних чатах** — у групі
  Telegram відхиляє таку клавіатуру. Тому «➕ Додати учасників» — кнопка-підказка.

---

## 7. Права

| Дія | Хто може |
|---|---|
| Додати/прибрати учасника (`+`/`-`) | будь-який **учасник** вечірки |
| Редагувати / видалити витрату | **автор** витрати або **адмін** чату |
| ✅ Завершити вечірку | лише **організатор** |

## 8. Структура коду

```
bot/
  config.py      — конфіг з env (BOT_TOKEN, DB_PATH)
  splitter.py    — чиста логіка поділу й мінімізації переказів (тестова основа)
  db.py          — шар даних (aiosqlite): коди, seq, транзакції, known_users
  parsing.py     — розбір витрат/команд з message entities
  keyboards.py   — reply- та inline-клавіатури
  formatting.py  — рендеринг повідомлень (HTML)
  handlers.py    — маршрутизація: кнопки, текст, callbacks, вхід у групу
  main.py        — точка входу (polling) + middleware реєстрації користувачів
tests/           — splitter, parsing, db, визначення бенефіціарів (46 тестів)
deploy/          — systemd-юніт, backup.sh, DEPLOY.md
```

Розгортання й бекапи — у [deploy/DEPLOY.md](deploy/DEPLOY.md).
